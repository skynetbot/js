Descripción

Como propósito en asistir a Santillana a mejorar o implementar el proceso de estandarización de escritura de código al momento de desarrollar software. El propósito de esta propuesta es para mejorar los procesos que ya estén en pie como también poner a Santillana en una mejor posición al momento de adquirir futuros contratos con desarrolladores de software. El resultado de esta propuesta será el Manual de Políticas y Procesos al Escribir Código en el Desarrollo de Software en acomodo para Santillana en su ambiente corporativo, histórico y políticas existentes.

La forma general la cual este Manual es usado es uno el cual define en totalidad el proceso de escritura de código en Santillana. Cada nuevo documento implementara las prácticas descritas por este Manual estandarizadas para Santillana. El resultado de la implementación de las Políticas y Procesos al Escribir Código en el Desarrollo de Software reduce efectivamente el tiempo de adaptación de un desarrollador como también mejora la comunicación entre desarrolladores en las siguientes áreas:

  • Documentación
  • Estructuración estándar
  • Legibilidad
  • Claridad
  • Planes de mejora
  • Resolución de problemas y bugs

Introducción

Este documento define los estándares de escritura de código en el desarrollo de software. El contenido en este manual está actualizado hasta la fecha del 6 de octubre del 2015. Las practicas de uso serán aplicadas al lenguaje en el cual desarrolla independientemente del lenguaje que se detalle en los ejemplos.

Sangría y espacios en blanco

Usa sangría de 4 espacios o tabulador. Las líneas no deben finalizar en espacio. Todo documento debe terminar en línea nueva (\n) no en retorno (enter \r). Todos los documentos de texto deben terminar en un solo new line (\n).

Operadores

Todos los operadores binarios como +, -, =, !=, ==, >, etcétera llevaran un espacio antes y después del operador. Por ejemplo, foo = bar; en vez de foo=bar;. Los operadores unarios no llevaran espacio entre el operador y la variable. Por ejemplo, ++foo;.

Estructuras de control

Deben tener un espacio entre la palabra clave de control y el paréntesis de apertura. Usa llaves hasta en situaciones las cuales no sean técnicamente necesarias. La llave de apertura debe estar colocada en la misma línea de la declaración. La llave de cierre debe estar colocada con sangría al mismo nivel de la declaración de apertura.

Declaración if: 


        if (foo == bar) {
            return;
        } else if (foo != bar) {
            return;
        } else {
            return;
        }

Declaración switch: 


        Switch (foo) {
            case 1:
                bar1;
                break;
            default:
                break;
        }

Declaración do: 


        do {
            foo;
        } while (bar);

Extensión de líneas

En general, las líneas no deben exceder mas de 80 caracteres. Solo es permitido exceder este numero en nombres largos de funciones, definiciones de funciones y clases, declaraciones de variables, etcétera. 


            if (contactFormSection2StudentAgeFieldText ==
        accountFormContactsSectionStudentGradeAgeText) {
                foo;
                bar;
            }

Condiciones no deben estar acortadas en líneas múltiples.

Llamada de funciones

Las funciones deben ser llamadas sin espacio entre el nombre de la función, el paréntesis de apertura y el primer parámetro y su coma; espacios entre los parámetros después de la coma y sin espacio entre el ultimo parámetro el paréntesis de cierre y el punto y coma. 


        variable = foo(bar, bazz, quuz);

En caso de un bloque de llamadas incrementar la cantidad de espacios para mejorar legibilidad 


        variableFormAccountText = foo(bar);
        variable                = foo(bazz);

Declaración de funciones


        function(argument) {
            variable['text'] = 'Hola Mundo!';
            return variable[argument];
        }

Constructores


        foo = 'newClass';
        foo = new newClass();
        foo = new newClass(bar, quuz);

Concatenaciones

        string = 'foo' . 'bar';

Definición de variables

Las variables serán definidas con un nombre descriptivo que haga referencia a su asignatura. Al utilizar múltiples palabras para una variable debe separarlas utilizando la tecla de subrayar o capitalizando la primera letra de la siguiente palaba. Mantenga uniformidad al utilizar el método que escoja (no utilice ambos). Variables globales deben ser capitalizadas y sus palabras deben ser separadas con la tecla de subrayar. 


        contactValue = 'foo'; // ESTO NO
        
        contactValue = 'foo'; // NO
        contact_text = 'bar'; // NO USAR MULTIPLES METODOS
        
        contactFormTab1Section1NameFieldValue = 'foo'; // SI
        contactFormTab1Section1NameFieldText  = 'bar'; // SI
        
        DEFINE_BACKGROUND = '#FFFFFF'; // GLOBALES

Documentación

Cada archivo debe incluir documentación al respecto de el propósito del mismo al igual como una descripción de la intención del código y sus funciones. Problemas en funcionalidad deben estar documentados en el archivo así como también funciones y métodos a implementarse.

Cada función estará debidamente documentada antes de su declaración con la descripción de su propósito. Métodos a implementarse deben estar documentados con las palabras "TODO" en caso del código estar incompleto en su función.

Compiladores, minimizadores de código

El uso de compiladores no aporta ningún beneficio de desempeño en código el cual se interpreta directamente y no requiere ejecución de binario. El tamaño de un archivo de código es mínimo comparado con el ancho de banda promedio. Uso de tales herramientas esta descartado.
